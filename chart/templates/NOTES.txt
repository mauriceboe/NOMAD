1. JWT secret handling
   - The chart exposes a small `jwt` configuration block in `values.yaml`.
   - Preferred methods to provide the JWT secret (choose one):
     - Generate a random secret at install: `--set jwt.generateSecret=true`
     - Use an existing Kubernetes Secret: `--set jwt.existingSecret=my-k8s-secret` (key name controlled by `jwt.existingSecretKey`, default `JWT_SECRET`)
     - Provide the secret value directly (not recommended for production): `--set jwt.secret=your_secret`

2. Example usage:
   - Generate a random secret: `helm install trek ./chart --set jwt.generateSecret=true`
   - Use an existing secret: `helm install trek ./chart --set jwt.existingSecret=my-k8s-secret --set jwt.existingSecretKey=MY_KEY`
   - Provide secret value directly (dev only): `helm install trek ./chart --set jwt.secret=my-secret-value`

3. Precedence and notes
   - If `jwt.existingSecret` is set, it takes precedence over `jwt.generateSecret` and `jwt.secret`.
   - If `jwt.generateSecret` is true, the chart will create a Secret named `<release>-jwt-secret` containing a random 32-character value under the key `JWT_SECRET` (or the key from `jwt.existingSecretKey` if customized).
   - Avoid embedding production secrets in `values.yaml`; prefer `jwt.generateSecret` or an externally managed Kubernetes Secret.

4. Environment variables
   - The `ConfigMap` template holds non-sensitive configuration (see `additionalEnv` / `additionalEnvFrom`).
   - Sensitive values should come from `jwt.*` (above) or from `additionalEnvFromSecrets` which references existing Kubernetes Secrets.
   - Ensure `env.ALLOWED_ORIGINS` matches your ingress hosts when using a browser with CORS.
